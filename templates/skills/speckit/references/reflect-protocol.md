# REFLECT 協議

## Red-Green-Reflect 紀律

`/speckit.execute` 中，先定義驗證條件（RED），再實作（GREEN），然後 **REFLECT 為強制關卡**——每個 action 完成後必須立即寫入 `insights.md`（有發現寫內容，無發現寫 `No new insights`）。禁止批次補寫。

- 程式碼任務：RED = 測試定義；GREEN = 程式碼實作
- 設計任務：RED = spec.md 中的 Acceptance Scenarios；GREEN = 設計稿修改 + 截圖驗證

## 副作用發現強制追蹤

執行過程中發現的任何**既有問題**（linter errors、型別錯誤、設計缺失、技術債），MUST 在 REFLECT 階段先判斷來源，再提出對應的追蹤方案，**不得僅說「與本次無關」後略過**。

### 第一步：判斷問題來源

| 來源 | 定義 | 處理 |
|------|------|------|
| 本專案引入 | 本次 action 新增或修改的程式碼造成 | 必須修正，不可遺留 |
| 本專案既有 | 修改前已存在，與本次無關 | 提出選項讓使用者決定 |
| 上游已知問題 | 來自依賴的上游程式碼 | 僅提醒，視嚴重程度建議 |

### 第二步：依來源決定行動

**本專案引入**（必修）：
- 立即修正，不提供「暫不處理」選項

**本專案既有**（選擇性）：
- `[A]` 追加至當前任務的 `insights.md` + `plan.md`（作為 Tech Debt action）
- `[B]` 建立新任務 `specs/[###-name]/` 獨立追蹤
- `[C]` 記錄至 `plan.md` Tech Debt 區段（暫不處理）

**上游已知問題**（輕重分級）：
- **輕**（不影響 runtime）：僅在 `insights.md` 標註 `[上游已知問題]`
- **重**（影響功能或安全）：提出修補方案，由使用者決定

## 迭代追蹤

當已完成的 action 因回饋、審查或修正而**再次修改**時，MUST 在 `insights.md` 對應段落追加迭代紀錄（含修正原因、根本原因分析、教訓）。若涉及新設計決策或引入新問題，新增獨立段落（D-0XX 編號延續）。此規則不限於 `/speckit.execute`——任何導致任務產物變更的互動皆適用。
