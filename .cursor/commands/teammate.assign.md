## Outline

Goal: Convert actions from actions.md into GitHub issues for project management and team coordination.

### Execution Steps

1. **Setup**: Run `.teammate/scripts/bash/check-prerequisites.sh --json --require-actions --include-actions` from repo root and parse:
   - `FEATURE_DIR`
   - `AVAILABLE_DOCS`
   - For single quotes in args like "I'm Groot", use escape syntax: e.g 'I'\''m Groot'

2. **Extract Actions Path**: From the executed script, extract the path to **actions.md**.

3. **Verify GitHub Remote**:

   Get the Git remote:
   ```bash
   git config --get remote.origin.url
   ```

   > [!CAUTION]
   > ONLY PROCEED TO NEXT STEPS IF THE REMOTE IS A GITHUB URL

4. **Parse Tasks File**:

   Load actions.md and extract:
   - All actions with their IDs (S001, S002, ...)
   - Phase groupings
   - User story associations ([US1], [US2], ...)
   - Scenario links ([Verifies: @tag])
   - Parallel markers ([P])
   - Dependencies

5. **Generate GitHub Issues**:

   For each action in actions.md, create a GitHub issue with:

   #### Issue Title
   ```
   [ActionID] [Story] Description
   ```
   Example: `[S012] [US1] Create User model in src/models/user.py`

   #### Issue Body
   ```markdown
   ## Action Details
   
   **ID**: [ActionID]
   **User Story**: [US#]
   **Verifies**: [Scenario tags]
   **Parallel**: [Yes/No]
   
   ## Description
   
   [Full description from actions.md]
   
   ## Acceptance Criteria
   
   - [ ] Implementation complete
   - [ ] Linked scenarios pass (GREEN)
   - [ ] Code reviewed
   
   ## Related Scenarios
   
   [List of Gherkin scenarios this action verifies]
   
   ## Dependencies
   
   [List of actions that must complete before this one]
   
   ---
   
   *Auto-generated by `/teammate.assign`*
   ```

   #### Labels
   - `action` (for all)
   - `phase-N` (based on phase)
   - `story-USN` (based on user story)
   - `parallel` (if [P] marker present)
   - `priority-P1/P2/P3` (based on story priority)

6. **Create Issues via GitHub MCP**:

   Use the GitHub MCP server to create issues:
   
   > [!CAUTION]
   > UNDER NO CIRCUMSTANCES EVER CREATE ISSUES IN REPOSITORIES THAT DO NOT MATCH THE REMOTE URL

7. **Update Tasks File**:

   After successful issue creation, update actions.md:
   - Add issue number reference to each action
   - Format: `- [ ] S012 [P] [US1] [Verifies: @tag] Description (#123)`

8. **Update Active Context**（Memory Delta Protocol）:

   Update `.teammate/memory/active-context.md` using delta mode:
   - **覆寫 `## Current State`**：Phase: Deliver, Last Command: assign, Next Action: /teammate.execute
   - **追加 `## Session Log`**：`| [timestamp] | assign | [N] issues created | [GitHub project link if any] |`

9. **Report Completion**:

   Output:
   - Number of issues created
   - Links to created issues
   - Any actions that couldn't be converted (with reasons)
   - Suggested next command: `/teammate.execute`

## Issue Organization

### Phase-Based Milestones

Consider creating GitHub milestones for each phase:
- `Phase 1: Setup`
- `Phase 2: Foundational`
- `Phase 3: User Story 1`
- etc.

### Labels Structure

| Label | Purpose |
|-------|---------|
| `action` | All action issues |
| `phase-1`, `phase-2`, etc. | Phase grouping |
| `story-us1`, `story-us2`, etc. | User story grouping |
| `parallel` | Can run in parallel |
| `blocking` | Blocks other actions |
| `step-def` | Step definition action |
| `impl` | Implementation action |

### Dependencies

For actions with dependencies:
- Reference dependent issues in the body
- Use GitHub's "depends on #N" convention if using project boards

## Error Handling

- If remote is not GitHub: Report and exit without creating issues
- If MCP server unavailable: Report error with manual instructions
- If issue creation fails: Log error, continue with remaining actions

## Best Practices

1. **Run after `/teammate.plan`**: Ensure actions.md is complete
2. **Review before assign**: Check actions.md for accuracy
3. **Use milestones**: Organize issues by phase
4. **Link to feature branch**: Reference the feature branch in issues
